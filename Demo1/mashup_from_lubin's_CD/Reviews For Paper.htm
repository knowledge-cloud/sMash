Title Voronoi-based Navigation for Web Mashups
Masked Reviewer ID: Assigned_Reviewer_1

Question 
Summary of the paper's main contributions and impact The authors design a novel visualized mashup system SurfToMash.
A Voronoi diagram based model is proposed to efficiently recommend top-k APIs for users.
Moreover, a comprehensive approach to rank the top-k APIs is also presented.
These techniques are integrated to implement a Web application system SurfToMash.
Strong points of the paper (please number them S1,S2,..) S1: It is novel to model the Web mashup recommendation problem as the k-nearest neighbor problem in Voronoi diagram.
S2: The problem formulation is clear and complete.
Weak points of the paper (please number them W1,W2,...) W1: Definition 4.2 may not be suitable for all situations (will be elaborated in detailed comments).
W2: The description of the generation of Voronoi diagram is not so clear.
Originality High
Quality of Presentation Medium
Significance Medium

Overall Recommendation Weak Reject
Detailed comments (please number each point) 1. The key point of this paper is to model the Web mashup recommendation problem as the k-nearest neighbor problem in Voronoi diagram.
As mentioned in Section 4.2, Step 1~3 are the preprocessing steps.
This means that the Voronoi diagram is constructed offline.
What I wonder is that whether Definition 4.2 is not suitable for all cases.
Concretely Speaking, if the APIs of chosen are not connected in Voronoi diagram, using the center of these chosen APIs to calculate the distance may not be appropriate.
Since the top-k APIs are extracted mainly based on this definition, I regard that this is a significant problem.
The above-mentioned case is not discussed in this paper.
2. Regarding Definition 3.1, P(A|B) may not be equal to P(B|A).
As mentioned in the paper, this probability is derived from the repository collecting from Programmable Web and Yahoo! Pipes.
However, what we know is the APIs used in a mashup.
How could we know whether API A is followed by API B?
In contrast, whether API B is followed by API A?
Since the effectiveness of the proposed approach will be influenced much by the distance matrix of APIs, this details of generating the distance matrix should be clearly depicted.
3. In Step 3 of Section 4.2, although the construction of Voronoi diagram is derived from the previous work, the description of this part is not so clear.
How to iterate the divide and conquer process?
How to generate ML and MR?
Further description is needed to make this part more understandable.
4. In the beginning of Step 1 in Section 4.2, it is mentioned that: Our work shows that there are almost 4,000 links amongst about 100 APIs.
However, in 100 APIs, the largest number of possible connections is: C(100, 2) = 4950. (Choose 2 from 100)
About 80% of APIs have connections between each other.
Is there any problem about this statistic?
5. Some parts of Step 1 and Step 2 in Section 4.2 are not clearly stated.
In Step 1 (up-left side of page 5), why the function f(x) should satisfy those constraints?
In Step 2 (Proposition 4.2), what is g(.,.)?
On the other hand, in the last paragraph of Section 5, the settings of these parameters are missing.
6. As mentioned in Section 6.2.1: We gave the standard ..., and then took them as the seed to generate 100,000 APIs ...
It is not clear how this process is done.
7. In Algorithm 2 of Step 3 in Section 4.2, the details of this algorithm can be further described.
What does A(xi,yi) in line 2 stand for?
What are 'low' and 'high' in line 10?
It is recommended that more details can be described in the paragraph.
8. Figure 2 seems not so correct.
The up-left edge should be farther away from the center point.
9. There are some typos.
In Section 2.1, H(pi,pi) -> H(pi,pj).
In Definition 3.1, 'Containning' -> 'containing'.
In Proposition 4.2, be 'an' set of -> be 'a' set of

Question 
Summary of the paper's main contributions and impact The paper proposes a mashup system that recommends new Web services to be integrated with Web services that are already chosen by the user. The recommendation is based on the linkage likelihood in the past mashups (how many users adopt a link from Filckr to GoogleMap in their mashups). A Voronoi diagram is introduced in order to compute top-k recommendation efficiently. To do so, the linkage likelihood is converted into a metric space, and a set of Web service APIs is mapped on a 2-dimensional space based on multi-dimensional scaling. The efficiency and accuracy of recommendation are demonstrated through experiments.
Strong points of the paper (please number them S1,S2,..) S1: A prototype is implemented and tested through a user study.
Weak points of the paper (please number them W1,W2,...) W1: The metrics and techniques used in the paper are not new by themselves.
W2: The problem formulation is not specific to mashup, and it does not seem new.
W3: Experiments does not compare the proposed metrics with others.
Originality Low
Quality of Presentation Medium
Significance Low
How many attendees will be interested in a talk on this paper? average interest
Technical Depth Low
Quality of Experiments, Repeatability Medium
Do the experiments support the claims made in the paper and match the motivating examples? no
Overall Recommendation Reject
Detailed comments (please number each point) (1) From the title "Voronoi-based...," I would assume a use of Voronoi diagram in Mashup recommendation is a key contribution of the paper. However, as written in Section 7, use of Voronoi diagram to K-nearest neighbor is not new. I do not clearly see any additional contribution to the existing work in this aspect.
(2) The problem formulation based on the users' past mashup statistics is quite generic (e.g. customers who purchased A also purchased B). As a result, I do not see novelty in the problem itself.
(3) I would expect new findings when recommendation is applied specific to mashup. What kind of features should be considered? Which metric perform better? etc. It would be nicer if the paper conduct such a comparative study rather than just presenting one approach that is not fundamentally new in general.
(4) The description in the paper lacks accuracy in some parts. For instance, in order to map probability P(A|B), P(B|A) to a metric space, the paper describes properties of functions f and g, but I do not find the actual functions used in the implementation.
(5) In Section 5, new scoring metrics on API quality is suddenly introduced. How are they integrated into the discussion up to Section 4?
(6) A prototype WITHOUT recommendation is compared to Yahoo! Pipes through a user study -- what is the point of this study, if you don't evaluate recommendation?
(7) Experiments show precision scores of the proposed approach. It must be compared with some baseline metrics to demonstrate effectiveness of the proposed metrics.

Question 
Summary of the paper's main contributions and impact This paper proposed a method to find the top-k nearest neighbor APIs to help the user
find the related data API effectively when developing mashup applications. They developed a system, SurfToMash, which enables users to surf a data API web to mashup
select APIs. The system also provides user with clear guidance by trace-based
recommendation based on a Voronoi-based technique.
Strong points of the paper (please number them S1,S2,..) S1 The overall idea and the problem setting are novel. The authors propose the problem
of top-k APIs recommendation based on users’ chosen APIs.

S2 The analytic hierarchy process to derive the weight of the ranking function is effective.

S3 The visualization of the recommendation prototype is easy for readers’ understanding.
Weak points of the paper (please number them W1,W2,...) W1 The paper convert a real world problem into mathematical representation and
then try to provide a solution to the real world problem by manipulating the
mathematical model. However, while doing so, the authors do not really look
back to see whether these mathematical manipulation make sense in real world.
At many points in time, we start asking whether such complex mathematical
manipulation is essential in solving the problem compare to more simple
solution.

W2 The paper does not really compare to other approaches in web mashups but
simply wave them aware as irrelevant Some of the comments are not true.
For example, the comments that [8] only solve Step 3 and 4 is not true since
[8] is about auto-complete of mashup and will thus also propose API to be connected
next in the mashup.

W3 The results of the basic and improved algorithm are totally different.
As such, it is not clear how the improved algorithm can be considered improved.
Furthermore, the authors haven’t compared the basic algorithm and the improved
one in the experiment section.

W4 The definition of the distance is based on the doubtful assumption. The
separation of distance and the ranking function seems a rather strange approach.

W5 There exists several mistakes in the paper.

Detailed comments (please number each point) The overall idea and the problem setting are good. The authors want to solve the
problem of top-k APIs recommendation based on users’ chosen APIs. Moreover, since they want to rank the k nearest neighbor with 4 attributes, they utilized the analytic hierarchy process to derive the weight of the ranking function. This is a
good way to handle multi-criteria decision making. Furthermore, the visualization of
the recommendation prototype is easy for readers to understand. However, the authors
need to consider the following issues to further improve the paper.

1. The definition 3.1 seems very similar to the confidence in association rule mining. The repository contains a set of rules, and we can model the problem as finding the top-k rules with maximum confidence. If the problem in this paper is different, the authors had better explain it more carefully.

2 The definition of the distance based on the doubtful assumption. P(A|C) >= P(A|B) *P(B|C) is reasonable for some cases, but may be not suitable for the general one. For example, P(A|C) may be equals to 0 since we cannot directly connect these two. But A could connect to C through B! More than that, this assumption tends to find the shortest connection between APIs which maybe not interesting. The longer path is the one that is hard to be composed by human. The separation of distance and the
ranking function is strange. The algorithm finds the top-k API based on distance
but the rank them by other ranking function which seems not so reasonable.

3 The results of basic algorithm and the improved algorithm are totally different, due to the different distance measure. The basic algorithm can be improved by pre-computing every P(A|B) to reduce the cost to the same level of Voronoi-based method, which also pre-computes every P(A|B). Furthermore, the basic algorithm can reduce the IO cost by TA based method. Right now it is assumed that the repository can be fit into memory, which is not the case. In the paper, the authors said it is a very complex network, and the API links is O(n*n). Using some index structure may be helpful. Lastly, the authors haven’t compared the basic algorithm and the improved one in the experiment section. The is no experiments done to convince
readers that the improved algortihm can beat the basic algorithm both in term of
time complexity and quality.


4 There exists several mistakes in the paper. For example, P(A|B) = P(B|A) =1 can only guarantee A,B appear in the same set of mashups, instead of A=B. Moreover, f(x) is the monotonic decreasing function, but d(A,B) may not be monotonic decreasing according to P(). Therefore, the top-k APIs cannot be viewed as the k-nearest neighbors of the chosen set. Also, the set of APIs user chosen may be next to
each other in a connected region of V.
